# symterm v1 Refined Plan

## Summary
实现两个 Go 二进制：本地 `symterm` 与远端 `symtermd`。`project` 表示一个共享的远端项目实例：它有且仅有一个共享工作区、一个 FUSE 挂载和一套运行态元数据，但允许多个本地 CLI 实例同时 attach，也允许在该共享工作区内并发运行多个命令。连接模型收敛为“单次 SSH 建立一个控制 channel 与零到多个 stdio channel”：控制 channel 承载控制面 RPC 与 FUSE 请求；每个被 attach 的 `command-id` 各自占用一个独立 stdio channel 承载 PTY 或 raw pipe 字节流。远端仅支持 Linux + FUSE3；本地支持 Windows、Linux、macOS，但文件系统语义收敛到可移植子集。

v1 的实例唯一键明确为 `<username>/<projectid>`。`username` 由 token 鉴权结果提供；`projectid` 由用户在同一 `username` 作用域内自行保证唯一。若多个本地目录错误复用了同一 key，系统仍允许 attach 到同一实例，但当来源差异超过阈值时必须把实例置为 `needs-confirmation` 状态：只允许只读 attach 与观察状态，禁止启动新命令、禁止增量同步，并把整个 project 临时切为只读；此后所有新的 `FsMutation` 与同步请求都必须被拒绝，已运行命令允许继续读取和退出，但任何后续写入都返回只读错误，直到用户显式确认“以当前本地目录为准重收敛”为止。远端文件永远不是权威状态；权威始终是当前 owner 客户端提供的本地文件视图。多个 CLI 可以并发连接同一个 key，共享同一个远端挂载目录；但同一时刻只能有一个 owner 持有同步权，其他客户端只能 attach、观察状态、启动命令和 attach 命令 stdio，不能并发驱动同步。

## Key Changes

### 1. 会话、身份与运行边界
- `symterm` 读取 `SYMTERM_ENDPOINT` 与 `SYMTERM_TOKEN`，再基于当前工作目录 basename 生成默认 `projectid` 并参与握手；若用户显式指定 `projectid`，则使用显式值。
- `symtermd` 使用 token 解析出的 `username` 与客户端提供的 `projectid` 组成实例 key：`<username>/<projectid>`。
- 远端项目根固定为 `~/.symterm/<username>/<projectid>`，内部至少拆分为：
  - `workspace/`：共享工作区，也是 FUSE 暴露给远端命令的目录。
  - `runtime/`：锁、PID、socket、同步暂存、挂载元数据等运行态文件。
  - `commands/<command-id>/`：具体命令的 PTY、退出码与附加元数据。
- 同一实例 key 只允许一个活动 `ProjectInstance`，但允许多个已 attach 的 CLI 客户端和多个并发运行的 `CommandInstance`。
- `ProjectInstance` 必须区分 owner 与 follower：
  - owner 独占同步权，负责首轮强收敛、增量同步、版本维护、FUSE 冲突校验和本地文件服务。
  - follower 只能 attach、观察状态、启动命令、attach 命令 stdio，不得并发发起同步。
- 新客户端 attach 到已有实例时，若其 `workspaceDigest` 或等价来源信息与当前实例记录明显不同，服务端必须评估来源差异等级：
  - 差异轻微时返回明确告警，但不改变当前实例状态。
  - 差异过大时把实例切换到 `needs-confirmation`，只允许只读 attach 与观察状态，禁止 `StartCommand`、禁止任何同步与写入；实例内已运行命令不被强杀，但其后续写入必须统一收到只读错误，直到收到显式确认的重收敛请求。
- 显式确认重收敛必须走单独 RPC：`ConfirmReconcile(projectID, expectedCursor, workspaceDigest)`。它只能在实例处于 `needs-confirmation` 时调用，并以 `expectedCursor` 作为前置条件，防止客户端对过期快照做确认；确认成功后，调用方成为新 owner，服务端生成新的 `syncEpoch` 并重新执行首轮强收敛。
- `symtermd` 生命周期与 `ProjectInstance` 绑定：鉴权、准备项目根、选举或确认 owner、首轮收敛、挂载 FUSE、接受 attach 与启动命令。首轮收敛失败、挂载失败或 owner 消失时，实例必须终止并清理运行态，不允许在未知同步状态下继续服务。

### 2. 首轮同步与文件级提交
- 首轮同步仍采用“元数据优先”：客户端发送 manifest，服务端只对可疑文件请求 hash；但整个收敛过程必须绑定一个显式的 `syncEpoch`。
- changed 文件的上传协议继续采用文件级原子提交，不允许裸 `ApplyChunk` 直接写目标路径。
- 控制面最小同步接口为：
  - `BeginSync(syncEpoch, rootFingerprint)`
  - `ScanManifest(entries)`
  - `NeedHashes(paths)`
  - `BeginFile(syncEpoch, path, metadata, expectedSize, statFingerprint)`
  - `ApplyChunk(fileID, offset, data, checksum)`
  - `CommitFile(fileID, finalHash, finalSize, mtime, mode, statFingerprint)`
  - `AbortFile(fileID, reason)`
  - `DeletePath(syncEpoch, path, precondition)`
  - `FinalizeSync(syncEpoch, rescanFingerprint)`
- `BeginFile` 只创建服务端临时文件句柄；所有 chunk 都写入临时文件。
- `CommitFile` 必须校验块连续性、最终大小、最终 hash，并以原子替换方式提升到目标路径；覆盖较短文件时以 commit 时的最终大小为准完成截断。
- 会话中断、校验失败或客户端显式取消时必须执行 `AbortFile` 或等价清理；未 commit 的半成品对远端命令和其他读者都不可见。
- owner 启动首轮同步时必须生成新的 `syncEpoch`，并保证同一轮里的 manifest、hash、文件内容与删除操作都绑定到该 `syncEpoch`。
- owner 在 `FinalizeSync(syncEpoch, rescanFingerprint)` 前必须执行一次最终复扫；若发现本轮扫描开始后本地文件继续变化，则把受影响路径标记为 dirty 并重试，或直接让本轮 `syncEpoch` 失败，不允许把来自多个本地时刻的混合状态发布为新的 `workspace/`。
- v1 同步对象仍只包含常规文件与目录；符号链接、设备文件、FIFO、不可表达路径、大小写冲突路径直接报错并拒绝启动。
- 删除策略保持“本地为准”，但删除范围严格限定在 `workspace/` 内；`runtime/`、`commands/` 等运行态目录绝不能参与收敛删除。
- 每次 `ProjectInstance` 首次成为 active、或 owner 显式请求重收敛时，都必须先把 `workspace/` 强制收敛到 owner 当前本地视图，再允许任何命令在该共享目录中启动。
- 若已有 follower attach 到同一实例，收敛完成前它们可以观察状态，但不得在旧快照上启动新命令。

### 3. FUSE、一致性与冲突校验
- `symtermd` 实现最小 FUSE 操作集：`lookup/getattr/readdir/open/read/write/create/mkdir/remove/rmdir/rename/truncate/chmod/utimens/fsync/flush/release`。
- 一致性模型保持“本地优先”，远端读取优先从 owner 提供的本地真实文件内容服务；daemon 可缓存页数据，但缓存必须受对象/目录代次变化和 owner 发出的 `Invalidate(changes[])` 驱动失效。
- 写入语义明确为“句柄本地可见”：
  - 同一已打开句柄必须能读到自己 staged 的写入内容。
  - 其他新打开的读者在提交成功前仍只能看到旧的已提交内容。
- staged 写入在 `fsync` 和 `flush` 时都必须尝试提交；普通程序即使不调用 `fsync`，在 `close(fd)` 触发的 `flush` 后也必须生效。
- 冲突在提交点暴露：`write()` 可以先返回成功，但若 `fsync/flush` 的提交因前置条件失配失败，必须向调用方返回明确冲突错误；对普通程序而言，这意味着 `close(fd)` 路径上的 `flush` 也必须稳定暴露该错误，不允许静默覆盖。
- `flush` 是 `close(fd)` 路径上提交成功或失败的主要可观察点：提交冲突、owner 落盘失败、版本推进失败等错误必须优先通过 `flush` 返回给调用方，而不是依赖 `release`。
- `release` 只负责句柄引用计数归零后的资源清理，不承担唯一提交语义；若仍有未提交脏数据，`release` 最多做幂等兜底清理或提交尝试，但不能作为唯一的错误回传路径。
- 每个已打开句柄都必须维护显式的 staged 状态机，例如 `clean/dirty/committing/committed/failed`；同一底层打开文件因 `dup`、多 fd 或多次 `flush` 导致的重复提交必须保证幂等，不得重复落盘或吞掉首个提交错误。
- `FsMutation` 的成功语义必须定义为“owner 已将本次变更写入本地文件系统并完成必要的持久化，然后推进受影响对象/目录的代次，最后才向远端回 ACK”；daemon 本地暂存成功、或仅在内存中记账，都不构成成功提交。
- 远端写入必须采用强同步提交路径：`PrepareMutation -> ApplyOnOwner -> OwnerAckCommitted -> PublishCommittedView`。只要 owner 落盘失败、`fsync` 失败、原子替换失败、版本推进失败或 owner 在确认前断开，远端都必须维持旧的 committed 视图，并向调用方返回明确错误。
- owner 客户端只维护 v1 所需的最小冲突元数据：被直接修改对象的 `objectGeneration`，以及其父目录的 `dirGeneration`。任何本地真实文件变更、目录项变更、rename、remove 都只推进受影响对象和父目录的代次。
- 冲突校验保持最小化：
  - `FsMutation(op, request, preconditions[]) -> FsReply` 只要求携带本次操作直接触及的对象与父目录前置条件，不要求构建通用路径版本树。
  - 前置条件默认使用 `objectGeneration` / `dirGeneration`；只有 `create/mkdir` 的“不存在断言”和 `rename overwrite` 的“目标仍是原对象”这类 v1 必需场景才额外校验对象身份，其他拿不准的情况直接返回冲突，让调用方重试。
- `Invalidate(changes[])` 只要求覆盖被改对象路径及其父目录，并能区分 `data`、`dentry`、`delete`、`rename` 这几类最小失效原因；negative dentry 等更细粒度缓存命中优化不作为 v1 必做项。
- `create/mkdir` 必须能表达“目标原本不存在”的前置条件；`rename overwrite` 必须校验目标是否仍是原对象，但这两类校验应作为少数特例处理，而不是把所有操作都扩展成通用多对象事务模型。
- follower 不维护独立文件监听、版本树或页缓存，只复用 `ProjectInstance` 的共享状态。
- 若 owner 本地文件监听短暂失效，实例必须切换为保守模式：停止信任已有缓存并对相关路径强制重新校验，而不是继续按旧版本服务。
- 若 owner 与 `ProjectInstance` 的控制连接断开，v1 直接终止整个实例：拒绝新的 attach，强制终止所有正在运行的命令，卸载 FUSE 并清理运行态，而不是尝试 owner handoff。
- 跨平台文件系统语义收敛到可移植子集：保证内容、目录结构、mtime；权限仅保留可移植部分，Windows/macOS 上不可表达的 POSIX 语义不做模拟，改为受控降级或显式失败。

### 4. 终端、PTY 与 attach-stdio
- 传输层固定为“1 个控制 channel + N 个 stdio channel”：控制 channel 负责 project 级 RPC、FUSE 请求与事件流；每次 `AttachStdio(commandID)` 都新开一个独立 stdio channel，绑定一个具体的 `command-id`。v1 明确采用“PTY 优先，raw pipe 回退”，但不在单个 stdio channel 内做多命令流复用。
- 远端命令入口在 `symtermd` 中全局唯一配置，不按 project 配置多套命令模板。
- 启动远端命令时，客户端只传本次调用的 `argv tail` 与 `ttySpec`；服务端将该 `argv tail` 原样拼接到全局配置的远端入口后执行。本地 CLI 参数与远端入口参数必须用 `--` 明确分隔：`symterm [local-options] -- <argv tail>`；`--` 之前只属于 `symterm`，之后原样进入 `StartCommand`。
- 启动远端命令时，若本地终端可交互，则服务端为子进程分配 PTY；attach 通道承载 stdin/stdout/stderr 字节流、窗口尺寸同步和信号转发。
- 命令控制统一收敛到显式的 `commandID` 寻址 RPC：控制面负责按 `commandID` 路由控制指令，stdio channel 只承载该命令的 PTY 或 raw pipe 字节流，不承担命令寻址语义。
- 控制面最小终端接口为：
  - `StartCommand(projectID, argvTail, ttySpec)`
  - `AttachStdio(commandID)`
  - `ResizeTTY(commandID, cols, rows)`
  - `SendSignal(commandID, name)`
  - `WatchCommand(commandID) -> live event stream`
- `StartCommand` 由任意已 attach 的客户端发起，但命令的工作目录固定为共享的 `workspace/`；命令之间共享文件视图，但各自拥有独立 `command-id`、PTY 和退出状态。
- `AttachStdio(commandID)` 打开的 stdio channel 在生命周期内只绑定一个命令；服务端不得把该 channel 上的控制帧或字节流路由到其他命令。
- `ResizeTTY` 与 `SendSignal` 必须显式携带 `commandID`，服务端仅对当前 project 中仍存活且调用方有权观察的命令执行控制；目标命令不存在、已退出或调用方不可见时必须返回确定错误。
- `ExitStatus` 不再作为客户端请求；命令退出、TTY 关闭、exec 失败等状态变化统一由 `WatchCommand(commandID)` 实时推送，并可被 `WatchProject(projectID)` 汇总转发为 project 级命令状态变更事件。断线重连后是否已退出、退出码等最终状态以 `EnsureProject` 返回的 `commandSnapshots` 为准，不要求 `WatchCommand` 提供游标续传。
- 同一个客户端若要同时观察多个命令，必须同时持有多个 stdio channel；例如观察 3 个命令时，连接形态为 `1 control channel + 3 stdio channels`。
- 非交互场景可退回 raw pipe，不强制 PTY；但交互式命令、彩色输出、Ctrl+C、全屏程序的测试目标只在 PTY 模式下承诺。
- CLI 不得在 attach 通道插入任何装饰输出；控制面与各命令的 stdio 数据必须物理隔离在不同 SSH channel 中。

### 5. 平台边界、状态接口与配置
- 远端仅支持 Linux + FUSE3，且具备用户态挂载能力。
- 本地支持 Windows、Linux、macOS；路径规范化、非法文件名检查、大小写碰撞检查在客户端先做。
- project 级配置尽量最小化，不承载命令入口或项目特定模板；project 主要用于实例 identity、工作区承载和运行态隔离。
- token 仅用于 daemon 会话鉴权，不参与 SSH 登录。
- `EnsureProject` 必须直接返回初始快照，包括 `projectID`、`role`、当前 project 状态、当前命令列表、是否允许启动新命令、当前 `syncEpoch`、是否处于 `needs-confirmation`、来源差异告警，以及生成该快照时对应的 `currentCursor`。
- `ProjectInstance` 必须维护单调递增的 project 级事件游标；任何会影响 attach 客户端可观察状态的变化，例如 owner 变化、收敛状态变化、命令状态变化、实例终止，都必须先分配新的 `projectCursor`，再发布快照或事件。
- 控制面新增统一 `WatchProject(projectID, sinceCursor)` 事件流，用于推送：
  - owner 变化
  - `needs-confirmation` 状态变化
  - 首轮收敛进度与完成状态
  - 命令启动、退出和状态变化
  - 实例终止事件
- `WatchProject(projectID, sinceCursor)` 返回的每个事件都必须携带自己的 `cursor`；服务端必须保证“先拿 `EnsureProject` 快照，再用返回的 `currentCursor` 建立 `WatchProject`”不会丢失快照之后发生的状态变化。
- 若 `sinceCursor` 过旧、未知或对应事件已被裁剪，`WatchProject` 必须返回明确的 `cursor expired` 类错误，要求客户端重新执行 `EnsureProject` 获取新快照，而不是静默从最新位置开始。
- 对命令级实时状态再提供 `WatchCommand(commandID)` live event stream，至少覆盖：stdio attach/detach、TTY resize 生效、信号送达结果、退出状态、exec 失败与 I/O 终止原因；它只负责当前连接上的实时观测，不承担断线续传，重连后由 `EnsureProject` + `WatchProject(projectID, sinceCursor)` 恢复 project 视角状态。
- follower 不依赖轮询推断状态；重连后先拿 `EnsureProject` 快照及 `currentCursor`，再接 `WatchProject(projectID, sinceCursor)` 增量事件。

### 6. 运维管理面、用户与登录策略
- `symtermd` 除业务控制面外，新增独立 admin plane，用于会话观测、用户管理、token 签发、策略配置与 Web 管理界面。admin backend 默认只监听远端主机本地 Unix Domain Socket：`~/.symterm/admin.sock`；不与业务控制 channel 复用，不直接暴露到公网。
- 管理界面的主要用户入口是 Web，而不是 CLI。CLI 继续保留，主要用于脚本化运维、应急调试和无浏览器场景。
- Web 管理界面由 `symtermd` 内建提供，默认监听独立 HTTP 地址，例如 `127.0.0.1:6040`；不与业务 listener 复用。推荐运维通过 SSH 端口转发访问浏览器页面，而不是直接把该端口暴露到公网。
- admin backend 的授权默认依赖本地 socket 文件权限与宿主机登录身份：能登录到远端并有权访问该 socket 的运维用户，才能执行管理动作。
- Web 管理界面必须有显式 admin 登录态；不能仅因为能访问 HTTP 端口就获得管理权限。若未来需要远程暴露 admin plane，必须走独立配置并增加显式 admin 鉴权，不与普通用户 token 混用。
- 业务会话与管理会话分离：普通 `symterm` 客户端只能访问业务 RPC；admin CLI 与 Web UI 只能访问 admin RPC。禁止通过现有 `Hello` / `EnsureProject` 链路做用户创建、token 发放或全局 session 枚举。
- Web 管理界面默认复用 admin backend，而不是直接读取业务内存结构；页面所见信息必须来自 admin API 与 admin 事件流，避免再做一套旁路读状态逻辑。
- `symtermd` 必须维护独立的可观测会话模型 `LiveSession`，它是用户可见的管理对象，不直接暴露内部 `clientID` 作为唯一管理键。最少字段包括：
  - `sessionID`
  - `clientID`
  - `username`
  - `projectID`
  - `role`
  - `transportKind`，至少区分 `tcp` / `ssh-tcp-forward` / `exec-local`
  - `remoteAddr`
  - `connectedAt`
  - `lastActivityAt`
  - `controlBytesIn` / `controlBytesOut`
  - `stdioBytesIn` / `stdioBytesOut`
  - `attachedCommandCount`
  - `projectState`
  - `syncEpoch`
  - `needsConfirmation`
- `remoteAddr` 必须在 transport accept / dial 边界采集，而不是由客户端自报。对于 `exec-local` 或无法可靠获得对端地址的场景，必须显式标记为 `local` / `unknown`，而不是伪造 IP。
- `session traffic` 统计按连接方向累计字节数，至少覆盖控制面与 stdio 两类。v1 不要求细分到每条 RPC，但必须保证：
  - 能看到当前 live session 的累计收发字节
  - 会话关闭前能拿到最终累计值
  - 多个 stdio channel 的流量会归并到所属 `LiveSession`
- admin 面看到的 `session sync state` 不再需要通过业务侧模拟推断，而是直接来自 project / session 当前快照，至少包含：
  - `projectState`
  - `syncEpoch`
  - `needsConfirmation`
  - 是否 owner
  - 最近一次同步进度快照
  - 当前运行命令数
- 用户管理从“启动时静态环境变量切分字符串”升级为持久化管理对象。远端新增 `admin/` 目录，至少包含：
  - `admin/users/<username>.json`
  - `admin/tokens/<token-id>.json`
  - `admin/audit.log`
- 用户对象 `UserRecord` 至少包含：
  - `username`
  - `disabled`
  - `createdAt`
  - `updatedAt`
  - `defaultEntrypoint`
  - `tokenIDs`
  - `labels` / `note`
- token 对象 `UserTokenRecord` 至少包含：
  - `tokenID`
  - `username`
  - `createdAt`
  - `lastUsedAt`
  - `revokedAt`
  - `description`
  - `secretHash`
- token 明文只允许在签发瞬间返回一次；磁盘只保存 hash，不允许回读原文。`获取 user token` 在管理语义上定义为“新签发一个 token”，而不是“把旧 token 明文取出来”。
- 现有 `SYMTERMD_STATIC_TOKENS` 收敛为 bootstrap / legacy 入口：
  - 首次部署或无持久化用户库时可用于导入初始用户
  - 一旦持久化用户库建立，静态 token 不再作为主要运营手段
  - admin 面必须能区分 `bootstrap token` 与 `managed token`
- 远端命令入口由“单个 daemon 全局入口”升级为“daemon 默认入口 + per-user 覆盖”：
  - daemon 级 `DefaultRemoteEntrypoint`
  - 用户级 `UserRecord.defaultEntrypoint`
  - 启动命令时优先使用用户级配置；若未配置则回退到 daemon 默认值
- per-user entrypoint 的作用域是“该用户登录后默认运行的远端入口模板”，不允许由普通业务客户端在每次握手时随意改写；修改必须走 admin plane。
- Web 管理界面必须提供实时反馈，不依赖固定轮询刷新。管理态实时事件统一收敛为 `AdminWatch` 流，并通过 WebSocket 暴露给浏览器。
- `AdminWatch` 至少覆盖：
  - session 新建 / 更新 / 断开
  - session traffic 增长
  - project sync state 变化
  - command 数量变化
  - 用户创建 / 禁用 / 更新
  - token 签发 / 吊销
  - per-user entrypoint 更新
- WebSocket 事件模型必须是“HTTP snapshot + WebSocket 增量”：
  - 首次打开页面先请求 HTTP snapshot
  - 再以 snapshot 携带的 cursor / version 建立 WebSocket 订阅
  - 若 cursor 过期，服务端返回明确错误并要求前端重拉 snapshot
- 管理前端不直接消费业务 `WatchProject` / `WatchCommand` 流；admin backend 负责把业务事件聚合成适合页面消费的管理事件，避免前端同时维护多套 cursor 和状态机。
- Web 管理界面最小页面结构为：
  - `/admin/`
  - `/admin/sessions`
  - `/admin/sessions/:sessionID`
  - `/admin/users`
  - `/admin/users/:username`
  - `/admin/daemon`
- Web UI 的写操作仍走普通 admin HTTP API；WebSocket 只负责服务端推送，不承载创建用户、吊销 token 这类命令型操作。
- admin plane 最小接口为：
  - `AdminGetDaemonInfo() -> version, startedAt, listenAddrs, adminSocketPath`
  - `AdminListSessions(filter) -> sessions[]`
  - `AdminGetSession(sessionID) -> session`
  - `AdminTerminateSession(sessionID, reason)`
  - `AdminListUsers() -> users[]`
  - `AdminGetUser(username) -> user`
  - `AdminCreateUser(username, disabled, defaultEntrypoint, note)`
  - `AdminDisableUser(username, disabled)`
  - `AdminIssueUserToken(username, description) -> tokenID, plainToken`
  - `AdminRevokeUserToken(tokenID)`
  - `AdminSetUserEntrypoint(username, argv)`
  - `AdminGetUserEntrypoint(username) -> argv`
  - `AdminWatch(cursor) -> event stream`
- Web 最小 HTTP / WebSocket 接口为：
  - `GET /admin/api/daemon`
  - `GET /admin/api/sessions`
  - `GET /admin/api/sessions/:sessionID`
  - `POST /admin/api/sessions/:sessionID/terminate`
  - `GET /admin/api/users`
  - `GET /admin/api/users/:username`
  - `POST /admin/api/users`
  - `POST /admin/api/users/:username/disable`
  - `POST /admin/api/users/:username/tokens`
  - `POST /admin/api/tokens/:tokenID/revoke`
  - `GET /admin/api/users/:username/entrypoint`
  - `POST /admin/api/users/:username/entrypoint`
  - `GET /admin/ws`
- CLI 侧新增管理命令族，默认通过 admin socket 访问，例如：
  - `symterm admin daemon info`
  - `symterm admin sessions list`
  - `symterm admin sessions inspect <session-id>`
  - `symterm admin sessions terminate <session-id>`
  - `symterm admin users list`
  - `symterm admin users create <username>`
  - `symterm admin users token issue <username>`
  - `symterm admin users token revoke <token-id>`
  - `symterm admin users entrypoint set <username> -- <argv>`
- admin plane 的写操作必须落审计日志，至少记录：
  - 管理动作类型
  - 操作者本地身份
  - 目标对象
  - 结果
  - 时间戳
- 当前实现收敛说明：
  - 统一鉴权结果已经落到 `AuthenticatedPrincipal`，业务侧内部仍用 `clientID` 寻址，但管理侧统一暴露 `sessionID`。
  - 管理侧会话聚合对象已实现为 `LiveSessionSnapshot`，统一包含 principal、transport metadata、control/stdio/ownerfs 字节累计、project 当前状态与终态关闭原因。
  - 管理持久化目录已落到 `admin/users/*.json`、`admin/tokens/*.json` 与 `admin/audit.log`；token 仅保存 hash，签发明文只在创建响应返回一次。
  - Web 管理端当前统一走 `HTTP snapshot + /admin/ws`，WebSocket 负责推送 cursor 化的全量快照；cursor 过期时页面回退到重新拉取 snapshot。
- v1 不引入复杂 RBAC；admin plane 默认是“宿主机运维权限即管理权限”。但必须为后续 RBAC 保留接口，不把“root 才能管”写死进业务协议。

## Test Plan
- 启动与握手：
  - 缺失 `SYMTERM_ENDPOINT` 或 `SYMTERM_TOKEN` 直接失败。
  - token 错误、远端无 `symtermd`、project 初始化失败时 fail fast。
  - `symterm [local-options] -- <argv tail>` 中，`--` 前的参数只由本地 CLI 解析，`--` 后的参数会原样进入远端入口；缺少 `--` 时不得把本地参数误传给远端命令。
  - 建立 `1 control channel + N stdio channels` 后，RPC 与各命令 stdio 无串流污染。
  - 同一 `<username>/<projectid>` 下多个 CLI 可并发 attach；其中仅一个客户端获得 owner 角色，其余获得 follower 角色。
  - 不同 `username` 绝不共享实例。
  - 同一实例 key 但不同本地来源 attach 时，轻微差异返回告警；结构差异过大时实例进入 `needs-confirmation`，且禁止 `StartCommand` 与同步写入，直到显式确认重收敛。
  - 实例进入 `needs-confirmation` 后，已运行命令不会被强杀，但其后续写入会收到只读错误；显式确认重收敛成功后，调用方获得 owner 并触发新的首轮同步。
  - `EnsureProject` 返回的 `currentCursor` 可直接作为后续 `WatchProject(projectID, sinceCursor)` 的起点；快照返回与事件流建立之间发生的状态变化不会丢失。
  - 使用过旧或未知的 `sinceCursor` 调用 `WatchProject` 时，会收到明确的游标过期错误，并能通过重新执行 `EnsureProject` 恢复。
  - 使用过期的 `expectedCursor` 调用 `ConfirmReconcile` 会被拒绝，避免基于旧快照错误接管实例。
- 首轮同步：
  - 新建、修改、删除、目录重命名、深层删除正确收敛。
  - 仅可疑文件触发 hash 补算，不做全量 hash。
  - 大文件分块上传可成功 commit；中断后无半成品可见。
  - 较短文件覆盖较长文件时最终大小正确，旧尾部被截断。
  - 重复块、漏块、块校验失败、最终 hash 不匹配都会 abort 且会话失败。
  - manifest、hash、上传内容与删除操作的 `syncEpoch` 必须一致；旧 epoch 的请求会被拒绝。
  - 首轮同步期间若 owner 本地继续变化，`FinalizeSync` 能检测到 dirty path，并触发重试或失败，而不会发布混合快照。
  - 符号链接、FIFO、大小写冲突路径、非法路径名触发启动失败。
  - 收敛只作用于 `workspace/`；`runtime/` 与 `commands/` 下文件不会被误删。
  - follower 不能发起首轮同步或增量同步。
- 会话期文件系统：
  - 远端创建、修改、删除、truncate、rename overwrite 都可回写到本地。
  - 同一打开句柄能读到自己 staged 的写入。
  - 其他新打开读者在 commit 前仍看到旧内容。
  - `fsync` 成功后新读者看到新内容。
  - 未调用 `fsync` 时，`close(fd)` 触发的 `flush` 仍会提交，并把提交冲突或 owner 落盘失败稳定返回给调用方。
  - `fsync/flush` 只有在 owner 本地已完成落盘并推进版本后才返回成功；若 owner 落盘失败，远端继续暴露旧的 committed 视图。
  - `release` 仅负责资源清理，不承担唯一提交语义；仅靠 `release` 不得丢失提交失败错误。
  - 本地先改、远端后写时，任何对象或父目录代次前置条件失配都返回明确冲突错误且旧权威内容保留。
  - `create` 的“不存在断言”和 `rename overwrite` 的对象代次校验行为稳定。
  - 源路径、目标路径、父目录并发变化时，直接触及对象与父目录的代次校验行为稳定。
  - `Invalidate(changes[])` 能驱动文件内容缓存与目录枚举缓存失效；更细粒度 negative dentry 优化不作为 v1 必测项。
  - 多个 CLI attach 到同一实例时，共享同一挂载目录和文件视图，但不会各自维护独立 watcher 或版本树。
  - 同一底层打开文件被 `dup`、多 fd 关闭或多次 `flush` 时，提交逻辑保持幂等；首次提交结果不会被后续重复 `flush/release` 覆盖或吞掉。
- 终端与生命周期：
  - `symterm -- --help` 可映射为远端全局入口程序的 `--help`，而本地 `symterm --help` 只显示本地 CLI 帮助。
  - PTY 模式下交互式命令、彩色输出、stderr 混合输出、窗口 resize、Ctrl+C 正确透传。
  - 非交互模式下 raw pipe 输出不被额外包装。
  - 同一实例下可并发启动多个命令；每个命令有独立 `command-id`、PTY、退出码和 attach-stdio 通道。
  - 同一客户端同时 attach 多个命令时，使用多个独立 stdio channel，而不是在单个 channel 内做多路复用。
  - `ResizeTTY(commandID, ...)` 与 `SendSignal(commandID, ...)` 在同一客户端并发控制多个命令时不会串到错误命令。
  - 对不存在、已退出或调用方不可见的 `commandID` 执行 `AttachStdio`、`ResizeTTY`、`SendSignal` 时返回确定错误。
  - `WatchCommand(commandID)` 能稳定推送 exec 失败、stdio attach/detach、退出状态和 I/O 终止原因；断线后通过 `EnsureProject` 与 `WatchProject` 恢复最终状态，`WatchCommand` 本身不承担按游标续传。
  - `EnsureProject` 返回完整初始快照与 `currentCursor`；`WatchProject(projectID, sinceCursor)` 能推送 `needs-confirmation`、收敛完成、命令启动、命令退出、owner 断开等事件，且快照到订阅之间不存在丢事件窗口。
  - 单个命令退出后不影响同一实例下其他命令与已 attach CLI。
  - owner 退出、本地文件服务崩溃、SSH 中断、FUSE 挂载失败时整个 `ProjectInstance` 被强制终止且不留僵尸挂载。
- 跨平台本地：
  - Windows、macOS、Linux 本地的路径规范化、mtime 同步、权限降级行为符合预期。
  - case-insensitive 文件系统遇到 case-only distinct 路径时报错，不静默覆盖。
- 管理面与用户：
  - admin socket 默认只允许本地访问，业务 listener 不暴露 admin RPC。
  - admin Web listener 与业务 listener 分离，默认仅绑定 `127.0.0.1`。
  - 浏览器打开管理页后，首屏通过 HTTP snapshot 渲染，后续变化通过 WebSocket 增量推送。
  - WebSocket 断开重连后，前端可基于 cursor 恢复；cursor 过期时会触发全量重拉而不是静默丢事件。
  - `AdminListSessions` 能看到 live session、`remoteAddr`、累计流量、当前 sync state 与命令数量。
  - `exec-local` / SSH 转发 / 原生 TCP 三类连接的 `transportKind` 与 `remoteAddr` 填充行为稳定。
  - session 断开后列表中移除，但 inspect / audit 中仍可看到终态统计。
  - 新建用户后可立即签发 token，并用该 token 建立业务会话。
  - token 明文仅在签发响应出现一次；磁盘与后续查询都只返回 metadata，不返回明文。
  - 吊销 token 后新连接被拒绝，已建立连接是否保留按 v1 规则明确且可测试。
  - 设置 per-user entrypoint 后，仅影响该用户后续新启动命令，不影响其他用户与已在运行命令。
  - 用户未设置 entrypoint 时回退到 daemon 默认入口；用户被禁用后新握手被拒绝。
  - admin 写操作全部落到 `audit.log`，包含操作者、目标对象、动作与结果。

## Assumptions
- v1 不支持符号链接、特殊文件、ACL、xattr、owner 映射。
- `projectid` 在同一 `username` 作用域内的唯一性由用户负责；冲突即共享同一实例。
- 不同来源复用同一实例 key 时，默认允许只读 attach；只有显式确认后才允许以当前本地目录为准重收敛。
- 远端命令入口全局唯一，不做 project 级模板分流。
- project 级配置最小化，不承载命令入口或额外策略。
- v1 admin 面默认不做复杂 RBAC，只依赖宿主机本地管理权限。
- v1 Web 管理界面采用内建静态资源 + 后端 HTTP/WS 服务，不要求单独前端部署链路。
- v1 token 管理不支持明文回读旧 token；只能新签发与吊销。
- v1 session 流量统计以进程内累计计数为准，不追求内核级网络统计精度。
- Windows 与 macOS 上 `chmod` 仅保留可移植部分，不承诺完整 POSIX 兼容。
- v1 不做 owner handoff；owner 消失时直接终止整个共享项目实例。
- 仓库初始化后按“协议层、共享项目实例层、同步/FUSE 层、命令/attach 层”四段实现，先打通 owner/follower 边界与冲突模型，再接文件系统与终端行为。
